## 2-1: 視点間相関検出

### 目的
同一の根本原因が複数の視点に影響している場合、個別指摘より優先度を昇格させる。

### 実行方法

Phase 1の結果を横断して以下のパターンを探す:

```
相関パターンの例:

「抽象化の欠如」
  → コード品質: 重複コード検出
  → テスタビリティ: モック困難
  → 保守性: 変更箇所が多い
  → 3つ以上で一致 → 根本原因: アーキテクチャ設計の問題 → CRITICAL昇格

「エラーハンドリング不足」
  → セキュリティ: エラー情報の漏洩
  → 可観測性: ログ不足
  → 信頼性: 障害時の動作不明
  → 3つ一致 → 根本原因: エラー設計の欠如 → HIGH昇格

「型設計の問題」
  → 型設計: any型/型アサーション
  → テスタビリティ: 契約不明確
  → ドキュメント: 振る舞いが不明
  → 2つ一致 → 根本原因: 型設計の見直し → MEDIUM→HIGH昇格
```

### 出力形式

```markdown
## 🔗 相関分析結果

### 相関グループ1: エラー設計の欠如（根本原因）
影響視点: セキュリティ, 可観測性, 信頼性（3視点）
昇格前: HIGH + MEDIUM + MEDIUM
昇格後: **CRITICAL**（複数視点への波及）
根拠: src/api/handlers.ts のエラーハンドリングが全体に影響
```

## 2-2: 赤チーム分析（攻撃・障害シナリオ）

### 目的
防御者（開発者）ではなく攻撃者・障害担当者の視点でシステムを見る。

### 攻撃者視点の問いかけ

```
1. 認証バイパスの最短経路は？
   - JWTの検証をスキップできる経路は？
   - セッション固定攻撃が可能な箇所は？
   - IDOR（Insecure Direct Object Reference）がある箇所は？

2. データ窃取の方法は？
   - SQLインジェクション可能な箇所は？
   - 認証なしにアクセスできるAPIは？
   - エラーレスポンスに内部情報が含まれているか？

3. サービス妨害の方法は？
   - レート制限がない高コストなエンドポイントは？
   - 無限ループや再帰的な処理が発生しうる箇所は？
   - メモリ/CPUの枯渇を引き起こせる処理は？

4. 内部脅威（悪意ある内部者）の行動は？
   - 通常ユーザーが管理者データにアクセスできるか？
   - 監査ログなしに本番データを変更できるか？
   - 誤操作で本番データを一括削除できるか？
```

### 障害シナリオの問いかけ

```
1. 「午前3時に壊れるとしたら？」
   - クーロンジョブやバッチ処理の失敗パスは？
   - タイムゾーン依存のバグは？
   - メモリリークによる長期間後の障害は？

2. 「外部APIが全滅したら？」
   - 外部API障害時のフォールバックは？
   - タイムアウト設定は適切か？
   - 外部APIへの依存箇所はサーキットブレーカーが必要か？

3. 「10倍トラフィックが来たら？」
   - データベース接続プールは十分か？
   - N+1クエリが爆発するシナリオは？
   - メモリ使用量が線形以上に増加するか？

4. 「データベースが一時停止したら？」
   - リトライロジックは適切か？
   - デッドロックが発生しうる箇所は？
   - トランザクションが正しくロールバックされるか？
```

### 出力形式


## 2-3: 進化適応分析（フラジリティスコア）

### 目的
コードが「時間とともに壊れやすくなっているか」を定量化する。

### gitログ分析

```bash
# フラジリティ候補ファイルTOP20
git log --since="90 days ago" --pretty=format: --name-only 2>/dev/null \
  | grep -v "^$" | sort | uniq -c | sort -rn | head -20

# バグ修正コミットの多いファイルを特定
git log --since="90 days ago" --pretty=format:"%s %H" \
  | grep -i "fix\|bug\|error\|hotfix" \
  | awk '{print $NF}' \
  | xargs -I{} git diff-tree --no-commit-id -r --name-only {} 2>/dev/null \
  | sort | uniq -c | sort -rn | head -10
```

### フラジリティスコア算出

```
フラジリティスコア = 変更頻度（90日）× 複雑度係数

複雑度係数の判定:
  ファイル100行未満, ネスト2以下 → ×1
  ファイル100-300行, ネスト3-4 → ×2
  ファイル300行超, ネスト5以上 → ×3
  多重責任（サービス+リポジトリ+変換ロジック混在）→ ×追加1

判定:
  スコア > 40 → 🔴 HIGH フラジリティ（即座にリファクタリング推奨）
  スコア 20-40 → 🟡 MEDIUM フラジリティ（計画的にリファクタリング）
  スコア < 20 → 🟢 LOW フラジリティ（許容範囲）
```

### 技術的負債トレンド算出

```bash
# 直近3ヶ月の技術的負債マーカー（TODO/FIXME/HACK）の増減
git log --since="90 days ago" --oneline -p 2>/dev/null \
  | grep "^+" | grep -i "TODO\|FIXME\|HACK\|XXX" | wc -l
git log --since="90 days ago" --oneline -p 2>/dev/null \
  | grep "^-" | grep -i "TODO\|FIXME\|HACK\|XXX" | wc -l
```

### 出力形式

```markdown
## 📈 進化適応分析

### フラジリティスコア TOP3
1. src/services/order.service.ts - スコア: 45（変更15回 × 複雑度3）🔴
2. src/api/products.ts - スコア: 28（変更14回 × 複雑度2）🟡
3. src/auth/middleware.ts - スコア: 21（変更7回 × 複雑度3）🟡

### 技術的負債トレンド
追加された技術的負債マーカー: +12件
解消された技術的負債マーカー: -4件
**純増: +8件/90日（⚠️ 増加トレンド）**

### 推奨アクション
order.service.ts を Phase 1（今週）でリファクタリング対象に指定
```

## 2-4: 不作為コスト（直さないとどうなるか）

### 目的
CRITICAL/HIGH問題を放置した場合の具体的な影響を時間軸で予測する。
「修正コスト」ではなく「放置コスト」を可視化することで優先度を明確にする。

### 各問題への問いかけ

```
今（現在）:
  - この問題がユーザーに影響を与えているか？
  - 現在のスケールでは顕在化していないか？

1ヶ月後（N倍スケール時）:
  - ユーザー数/データ量が2-3倍になったら？
  - 新機能追加時にこの問題がブロッカーになるか？
  - バグが本番で発生する確率は？

6ヶ月後（チームが変わった時）:
  - 新しい開発者がこのコードを修正できるか？
  - このコードが他の問題の根本原因になるか？
  - セキュリティ脆弱性として悪用される可能性は？
```

### 出力形式

```markdown
## 💸 不作為コスト分析

### [HIGH] N+1クエリ問題（src/repositories/product.ts:45）

**今**: 商品数1,000件では0.3秒で許容範囲
**1ヶ月後（商品10,000件）**: 3秒超 → ユーザー離脱率20%増加推定
**6ヶ月後（商品100,000件）**: タイムアウト → 機能停止

**修正コスト（6ヶ月後）**: 2週間（データ構造の見直しが必要）
**結論**: 今修正しないと修正コストが×10倍になる
```

## 2-5: ビジネス価値検証

### 目的
コードが「あるべきかどうか」を検証する。技術的に正しくても、価値のないコードは負債。

### 削除テスト

```
各モジュール・関数に対して問う:
「このコードを削除したら何が困るか？」

答えられない場合 → 削除候補
答えが曖昧 → 仕様書との照合が必要
「誰も使っていないはず」 → 実際に呼び出し元を確認
```

```bash
# 実際に呼び出されていない関数・クラスを特定
grep -r "functionName\|ClassName" src/ --include="*.ts" | grep -v "def\|export" | wc -l
```

### オーバーエンジニアリング検出

```
以下のパターンを検出:
1. インターフェースに実装が1つしかない
   → 抽象化の必要性を疑う（テスト目的は除く）

2. 設定で切り替えられるが実際には1パターンしか使われていない
   → 設定の簡略化を検討

3. 汎用ユーティリティが1箇所でしか使われていない
   → インライン化を検討

4. 中間変換層が増殖している（A→B→C→D変換）
   → 変換の統合を検討
```

### 仕様乖離検出

```
コードと要件定義書/READMEを比較:
- README: 「〜機能をサポートする」→ コードに実装なし → 未実装または削除済み
- README: 「〜は対応していない」→ コードに実装あり → デッドコードの可能性
- API仕様書のエンドポイント vs 実際のルート → 乖離の検出
```

### 出力形式

```markdown
## 🏗️ ビジネス価値検証

### 削除候補
- src/utils/legacy-formatter.ts — 呼び出し元: 0件（grep調査済み）
- src/adapters/old-api-adapter.ts — 呼び出し元: 1件（移行済みの可能性）

### オーバーエンジニアリング疑義
- IProductRepository インターフェースに実装が1つのみ（ProductRepository）
  → テスト目的でなければ直接クラス参照を検討
- config/feature-flags.ts の "experimental" フラグが全てfalse
  → フラグ自体を削除してコードをシンプルにする

### 仕様乖離
- README: 「マルチ通貨対応」→ コードにcurrencyフィールドが未実装
  → 設計不足か要件変更かを確認が必要
```

## 2-6: 変更容易性測定

### 目的
「将来の変更がどれだけ難しいか」を測定する。今動いているかより、来月も変更できるかが重要。

### 変更増幅係数（Change Amplification Factor）

```
測定方法:
1つの概念変更が何ファイルの修正を要求するか

例:
「注文ステータスにCANCELLED_BY_ADMINを追加する」
  → OrderStatus enum修正: 1ファイル
  → order.service.ts のswitch文: 1ファイル
  → order.repository.ts のフィルタリング: 1ファイル
  → admin.api.ts のハンドラ: 1ファイル
  → order.mapper.ts の変換: 1ファイル
  → email-templates/ の通知テンプレート: 3ファイル
  合計: 8ファイル → 変更増幅係数: 8

判定:
  係数 1-3 → ✅ 良好（凝集度が高い）
  係数 4-7 → 🟡 注意（分散しすぎ）
  係数 8+ → 🔴 問題（DRY原則違反、設計の見直しが必要）
```

### 変更シナリオテスト（3シナリオ固定）

```
シナリオ1: 「認証方式を変更したら何ファイル修正が必要か？」
  - 現在: JWT → 新方式: Session/OAuth
  - 影響ファイルを全てリストアップ
  - 修正方法: auth middleware / session設定 / フロントエンドのリクエスト

シナリオ2: 「外部APIのレスポンス形式が変わったら何ファイル修正が必要か？」
  - 最も重要な外部API（1〜2つ）のレスポンスが全て変わると仮定
  - 影響ファイルを全てリストアップ
  - 現在の型定義の変更がどこまで波及するか

シナリオ3: 「新しい決済方法を追加したら何ファイル修正が必要か？」（決済なし → 「DBを変えたら？」）
  - 現在の決済/DB方式に最も非互換な変更を想定
  - 変更が必要なファイルを全てリストアップ
```

### 出力形式

```markdown

### 変更増幅係数
注文ステータス変更: **8ファイル** 🔴（目標: ≤3）
  原因: ステータスがenum/service/repository/api/templateに分散

認証ユーザー情報の形式変更: **4ファイル** 🟡
  原因: UserContext型が複数ファイルで直接参照

### 変更シナリオ分析
| シナリオ | 修正必要ファイル数 | 判定 |
|---------|-----------------|------|
| 認証方式変更 | 12ファイル | 🔴 設計見直し推奨 |
| 外部APIレスポンス変更 | 6ファイル | 🟡 型定義の集約を検討 |
| DB変更（SQLite→Postgres） | 3ファイル | ✅ repository抽象化済み |

### 最も変更しにくい箇所
src/services/order.service.ts — OrderStatus, 外部API, メール送信が密結合
**推奨**: OrderStatus専用モジュールへの抽出
```

## 2-7: 認知負荷評価

### 目的
「次の開発者がこのコードを理解できるか」を測定する。バグがなくても理解できないコードは技術的負債。

### メンタルモデル複雑さの測定

```
Miller's Lawの適用（同時に頭に置くべき概念の上限: 7±2）

測定対象: 関数・クラス・モジュール
測定方法: コードを読み解くために必要な「前提知識」を数える

例:
function processOrder(order: Order, user: User, ctx: Context) {
  // このコードを理解するために必要な知識:
  // 1. Order型の構造（8フィールド）
  // 2. User型の権限モデル（役割3種類）
  // 3. Contextオブジェクトの使い方（5メソッド）
  // 4. 注文の状態遷移ルール（6状態）
  // 5. 外部APIの制約（2つ）
  // 6. メール送信のタイミングルール
  // 7. トランザクションの境界
  // 合計: 7つ → Miller's Law上限 → 要注意
}
```

### サプライズ原則違反（Principle of Least Surprise）

```
「このコードの振る舞いは名前・構造から推測できるか？」

検出パターン（必ず具体例を挙げる）:

【隠れた副作用】
  bad: save() の中で sendEmail() を呼ぶ
  bad: getUser() の中でユーザーのlastLoginAtを更新する
  bad: validateOrder() の中でDBにレコードを作成する

【逆の期待裏切り】
  bad: isLoggedIn() が副作用としてセッションを更新する
  bad: delete() が実際には soft delete（論理削除）する（コメントなし）
  bad: calculateTotal() が税込みか税抜きかわからない

【名前と実装の乖離】
  bad: updateUser() が実際にはcreate or updateする（upsert）
  bad: getProducts() が内部でcacheを更新している
  bad: validate() がtrueを返す場合もfalseを返す場合も何かを変更する
```

### 暗黙知の可視化

```
「このコードを正しく使うために知っていなければならない前提」を列挙:

例:
- この関数を呼ぶ前に必ずconnect()を呼ぶ必要がある → コメント/型で強制されているか？
- 引数のidxはゼロベースではなく1ベース → 変数名/コメントで明示されているか？
- このAPIは1秒に1回しか呼べない → レート制限チェックが実装されているか？
- transactionが進行中の場合この関数はスキップされる → ドキュメントにあるか？
```

### 出力形式

```markdown

### 認知負荷TOP3（高い順）
1. src/services/order.service.ts:processOrder — 前提知識数: 9（上限7を超過）
   → 3つのサブ関数に分割することで各6以下に削減可能

2. src/api/webhook.ts:handleShopifyEvent — 前提知識数: 8
   → イベントタイプごとにハンドラを分離することを推奨

3. src/repositories/product.ts:bulkSync — 前提知識数: 7（上限ギリギリ）
   → コメント追加で許容範囲になる可能性あり

### サプライズ原則違反: 3件
1. [HIGH] src/services/auth.service.ts:verifyToken — トークン検証と同時にlastAccessを更新（隠れた副作用）
2. [MEDIUM] src/repositories/order.ts:deleteOrder — soft deleteだが名前から分からない
```

## 2-9: SLOインパクト評価

### 目的
この変更が本番環境のSLO/SLAに与えるリスクを評価し、適切なデプロイ戦略を提言する。
障害が起きる「前」に影響範囲を明確にすることで、カナリアデプロイやフィーチャーフラグの使用判断を支援する。

### 評価フロー

```
問い1: この変更はリクエストのクリティカルパスに触れているか？
  YES → SLO直撃リスクあり → 慎重にデプロイ（カナリア推奨）
  NO  → 影響軽微（即時デプロイ可）

問い2: エラーレートが上昇するシナリオはあるか？
  - DB接続エラー時のフォールバックは実装されているか？
  - 外部API障害時に5xxを返す可能性がないか？
  - 新しいバリデーションが既存データを拒否しないか？

問い3: レイテンシーが増加するシナリオはあるか？
  - 新規DB呼び出しが追加されたか？（N+1の可能性）
  - 同期処理になった箇所がないか？（async/awaitの順序変更）
  - 新しい外部API呼び出しがあるか？

問い4: ロールバックは安全に可能か？
  - DBマイグレーションを含む変更は ロールバック = データ損失リスクがある
  - フィーチャーフラグで段階的リリースできるか？
```

### SLI（サービスレベル指標）影響チェックリスト

```
主要SLI:
  可用性SLI: この変更でサービスが停止するシナリオがあるか？
  レイテンシーSLI: P99レイテンシーが変わる可能性があるか？
  エラーレートSLI: 成功率が変わる可能性があるか？
  スループットSLI: スループットの上限が変わるか？

エラーバジェット消費評価:
  月次エラーバジェット: 99.9% SLO → 月43.8分の許容ダウンタイム
  この変更でデプロイ失敗が発生した場合、何分消費するか？
  現在のバジェット残高は十分か（月末に近い場合は慎重に）？
```
