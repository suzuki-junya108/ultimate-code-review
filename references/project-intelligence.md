# Phase 0: プロジェクト知性の獲得

レビューの文脈を理解し、適切な重み付けを決定するためのガイド。
プロジェクトタイプとフェーズを正確に判定することで、的外れな指摘を排除する。

---

## プロジェクトタイプ判定フロー

### ステップ1: ファイル構造スキャン

```bash
# 主要ファイルの確認
ls package.json tsconfig.json pyproject.toml Cargo.toml go.mod 2>/dev/null
ls -d src/ app/ lib/ api/ 2>/dev/null
```

### ステップ2: タイプ判定ロジック

```
package.json の dependencies を確認:

next / nuxt / gatsby / remix
  → タイプ: Webフロントエンド（フルスタック）
  → 重点: パフォーマンス★★★, アクセシビリティ★★★, SEO★★★

react / vue / svelte / solid（単体）
  → タイプ: SPAフロントエンド
  → 重点: バンドルサイズ★★★, アクセシビリティ★★★, i18n★★

express / hono / fastify / koa / workers / nest
  → タイプ: REST API / バックエンド
  → 重点: セキュリティ★★★★★, API設計★★★, 後方互換性★★★

外部公開されるindex.ts/index.js のみ（appなし）
  → タイプ: ライブラリ/パッケージ
  → 重点: 後方互換性★★★★★, ドキュメント★★★★, API設計★★★★

prisma / drizzle / typeorm / sqlalchemy / ent
  → タイプ: データ集約型
  → 重点: データモデル★★★★★, パフォーマンス★★★★, 信頼性★★★

auth / jwt / oauth / stripe / payment がある
  → タイプ: 認証・課金系（最高リスク）
  → 重点: セキュリティ★★★★★, テスタビリティ★★★★★, 監査ログ★★★★

commander / yargs / oclif / click / cobra
  → タイプ: CLI/ツール
  → 重点: オンボーディング★★★★, エラーハンドリング★★★, ドキュメント★★★
```

## 開発フェーズ判定フロー

### gitログ分析

```bash
# コミット数でフェーズを推定
git log --oneline | wc -l

# 直近の変更頻度
git log --since="30 days ago" --oneline | wc -l

# テストファイルの存在確認
find . -name "*.test.*" -o -name "*.spec.*" 2>/dev/null | grep -v node_modules | wc -l
```

### フェーズ判定ロジック

```
コミット数 < 30 かつ テストファイル < 5
  → フェーズ: PoC/プロトタイプ
  → 寛容にする: コンプライアンス, 後方互換性, ドキュメント
  → 厳しくする: セキュリティ基礎, 機能的正確性

コミット数 30-200 かつ CI/CDなし
  → フェーズ: MVP
  → 寛容にする: 認知負荷, 変更容易性
  → 厳しくする: セキュリティ, 信頼性, テスト基礎

コミット数 > 200 または CI/CDあり
  → フェーズ: 本番運用
  → 全視点を最大厳格度で適用

PRがリファクタリング目的（コミットメッセージにrefactor/cleanupが多い）
  → フェーズ: リファクタリング
  → 重点: アーキテクチャ, 複雑度削減, DRY
  → 軽視OK: 機能的正確性（既存動作の維持を前提とする）
```

## ホットスポット分析

変更頻度が高いファイル = フラジリティリスクが高い = 重点レビュー対象

```bash
# 直近90日の変更頻度TOP20
git log --since="90 days ago" --pretty=format: --name-only 2>/dev/null \
  | sort | uniq -c | sort -rn | head -20
```

### フラジリティスコア算出

```
フラジリティスコア = 変更頻度（直近90日）× 推定循環複雑度

推定循環複雑度の目安:
- 100行未満の関数型コード → 低（×1）
- 100-300行, if/switch多用 → 中（×2）
- 300行超, 深いネスト → 高（×3）

フラジリティスコア > 20 → Phase 1で重点的に分析
フラジリティスコア > 40 → Phase 2でも個別言及
```

## 重み付けマトリクス

### タイプ別重点視点

| プロジェクトタイプ | ★★★★★ | ★★★★ | ★★★ |
|-----------------|-------|------|-----|
| Webフロントエンド | アクセシビリティ, パフォーマンス | i18n, セキュリティ | テスト, ドキュメント |
| REST API | セキュリティ, API設計 | 後方互換性, 信頼性 | パフォーマンス, ドキュメント |
| ライブラリ | 後方互換性, API設計 | ドキュメント, テスト | 型設計, パフォーマンス |
| データ集約型 | データモデル, パフォーマンス | 信頼性, セキュリティ | テスト, マイグレーション安全 |
| 認証・課金系 | セキュリティ, 監査ログ | テスト, エラーハンドリング | 後方互換性, 信頼性 |
| CLI/ツール | UX/エラーメッセージ, ドキュメント | オンボーディング, 信頼性 | テスト, パフォーマンス |

### フェーズ別厳格度調整

| フェーズ | 追加する厳しさ | 軽減する厳しさ |
|---------|-------------|-------------|
| PoC | セキュリティ基礎, データ漏洩防止 | i18n, アクセシビリティ, CI/CD |
| MVP | セキュリティ全般, 基本テスト | 認知負荷の細部, 変更容易性 |
| 本番 | 全視点最大 | なし |
| リファクタリング | アーキテクチャ一貫性 | 機能追加的な指摘 |

## 変更対象の分類

Phase 1で使用する変更ファイルを分類する:

```bash
git diff --name-only HEAD~1 HEAD 2>/dev/null \
  || git diff --name-only --cached \
  || git diff --name-only HEAD
```

### 分類ルール

```
**/auth/** **/security/** → セキュリティ視点に★追加
**/payment/** **/billing/** → セキュリティ+監査ログに★追加
**/api/** **/routes/** → API設計+後方互換性に★追加
**/*.test.* **/*.spec.* → テスタビリティ視点のみ
**/migrations/** **/schema.* → データモデル視点に★追加
**/config/** **/.env* → 設定管理視点に★追加
package.json / **/deps → 依存関係視点に★追加
**/*.md **/docs/** → ドキュメント視点のみ（他は軽量）
```
