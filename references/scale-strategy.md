# スケール戦略リファレンス

変更ファイル数が30件を超えた場合に自動適用されるチャンク完全分析プロトコルの定義。
品質は一切落とさない。全ファイルを29視点で完全レビューする。

---

## 1. スケール検出プロトコル

### ステップ 0-a: 引数解析

スキル起動時の引数から以下のオプションを抽出する:

```
--path <dir>       サブディレクトリフィルタ（複数指定可）
--package <name>   モノレポパッケージ指定（複数指定可）
--base <ref>       git diff のベースリビジョン（デフォルト: HEAD~1）
--staged           git diff --cached を使用（ステージング済みのみ）
```

引数解析の優先順位:
1. `--staged` が指定された場合: `--base` は無視し `git diff --cached` を使用
2. `--base` が指定された場合: `HEAD~1` の代わりに指定値を使用
3. `--package` が指定された場合: パッケージ名をディレクトリパスに解決してから `--path` として扱う
4. `--path` が指定された場合: ファイルリストをプレフィックスフィルタリング

### ステップ 0-b: 変更ファイルリスト取得

```bash
# 通常（--staged なし）
git diff --name-only "${BASE:-HEAD~1}" HEAD 2>/dev/null \
  || git diff --name-only --cached 2>/dev/null \
  || git diff --name-only HEAD

# --staged 指定時
git diff --name-only --cached
```

`--path` フィルタの適用（複数指定はOR条件）:
```bash
# 例: --path src/auth --path src/payment
# 結果: src/auth/ または src/payment/ で始まるファイルのみ
```

フィルタ後に 0 件だった場合は F-7 回復手順を実行する。

### ステップ 0-c: ファイル数カウントとスケール判定

```
SCOPE_FILE_COUNT = フィルタ適用後の変更ファイル数

SCOPE_FILE_COUNT ≤ 30  → 通常モード（チャンク分析なし）
SCOPE_FILE_COUNT > 30  → チャンク完全分析モード
```

### ステップ 0-d: モノレポ検出

以下のファイルをプロジェクトルートで確認する（先にマッチしたものを採用）:

| シグナル | ツール |
|---------|--------|
| `nx.json` | Nx workspace |
| `turbo.json` | Turborepo |
| `pnpm-workspace.yaml` | pnpm workspaces |
| `lerna.json` | Lerna |
| `rush.json` | Rush |
| `package.json` に `"workspaces"` キー | npm/Yarn workspaces |
| `Cargo.toml` に `[workspace]` セクション | Cargo workspace |
| `go.work` ファイル | Go workspaces |
| `packages/` または `apps/` ディレクトリに複数の `package.json` | 慣例的モノレポ |

モノレポ検出時のパッケージ一覧取得:

```bash
# pnpm-workspace.yaml の場合
cat pnpm-workspace.yaml  # packages: グロブを読む

# Cargo workspace の場合
grep -A 20 '\[workspace\]' Cargo.toml | grep 'members'

# go.work の場合
grep '^use ' go.work | awk '{print $2}'

# package.json workspaces の場合
node -e "console.log(JSON.parse(require('fs').readFileSync('package.json','utf8')).workspaces?.join('\n'))" 2>/dev/null
```

`--package api` が指定された場合:
1. 上記コマンドでパッケージ一覧を取得
2. `api` にマッチするディレクトリパスを特定
3. そのパスを `--path` として適用
4. マッチしない場合は F-8 回復手順を実行

### ステップ 0-e: チャンク計画

```
CHUNK_SIZE = 20

NUM_CHUNKS = ceil(SCOPE_FILE_COUNT / 20)
```

**グループ化優先順位**（品質向上のため、関連ファイルを同一チャンクにまとめる）:

1. 同一モノレポパッケージ（`packages/api/` 配下は同一チャンクに優先）
2. 同一ディレクトリ（`src/auth/` 配下は同一チャンクに優先）
3. 同一レイヤー（routes / services / models 等）

チャンク割り当て手順:
```
1. ファイルリストをパス順（ディレクトリ構造が近いもの順）でソート
2. 上から20件ずつ切り出してチャンクを作成
3. 各チャンクに番号付け: チャンク 1/N, 2/N, ...
```

### ステップ 0-f: 所要時間告知

ユーザーに処理開始前に告知する:

```
[SCALE] スコープ: N ファイル → M チャンク で完全分析を実行します。

チャンク数 | 推定時間
---------|--------
1        | 25-35分
2        | 45-60分
3        | 65-90分
5        | 100-140分
10       | 190-260分

推定所要時間: XX-YY分

スコープを絞るには: /review --path <ディレクトリ>
```

## 2. チャンク完全分析プロトコル

### Phase 1 チャンクループ

各チャンクに対して、**現行と全く同じ29視点フル分析**を実行する。

```
チャンク 1/M を処理中（ファイル 1-20）
  [対象ファイル一覧]
  → 5エージェント×29視点の完全分析を実行
  → findings を保存: chunk_1_findings

チャンク 2/M を処理中（ファイル 21-40）
  [対象ファイル一覧]
  → 5エージェント×29視点の完全分析を実行
  → findings を保存: chunk_2_findings

...

チャンク M/M 処理完了
```

各チャンクへのエージェント指示テンプレート（現行と同じ形式を維持）:

```
あなたはコードレビュー専門エージェントです。
以下の視点でコードを分析し、構造化された結果を返してください。

[チャンク N/M 処理中]
対象ファイル: [このチャンクの20件以下のファイルリスト]
プロジェクトタイプ: [Phase 0で判定したタイプ]
フェーズ: [Phase 0で判定したフェーズ]
累積findings: [前チャンクまでの重要問題サマリー（前チャンクがある場合）]

分析する視点: [カテゴリ名と視点番号リスト]
詳細ガイド: [該当するperspective-*.mdの内容]

※ 前チャンクと同じファイルは含まれていません。各ファイルを同じ深度で分析してください。

出力形式:
## カテゴリ[A/B/C/D/E]分析結果（チャンク N/M）

### 視点[N]: [視点名]
評価: ★☆☆☆☆ ~ ★★★★★
[CRITICAL/HIGH/MEDIUM/LOW] [問題の説明] - [ファイル:行番号]
  根拠: [具体的なコード/パターン]
  修正案: [具体的な修正方法]

良い点: [ポジティブな観察]
```

### Phase 1 findings 結合

全チャンク完了後、findings を結合する:

```
ALL_FINDINGS = chunk_1_findings + chunk_2_findings + ... + chunk_M_findings

重複排除:
  同一 (ファイル, 行範囲, 視点番号) の重複エントリを排除
  より高い重大度のエントリを残す

チャンク間相関:
  異なるチャンクのファイル間で同一根本原因が疑われる問題をマーク
  → Phase 2 の 2-1（視点間相関検出）の追加入力として渡す
```

### Phase 1.5 チャンク対応

Phase 1.5 は全チャンクの分析完了後、ALL_FINDINGS 全体のファイルリストを対象に**1回だけ**実行する（チャンク分割しない）:

```
対象ファイル: 全変更ファイル（全チャンクの合計）
実行方法: 現行と同じ（主要言語: 全カテゴリ / 副次言語: CRITICAL/HIGH のみ）
```

### Phase 2 チャンク対応

Phase 2 は ALL_FINDINGS（全チャンクの結合済みfindings）を入力として**1回だけ**実行する:

- 2-3（進化適応分析）は `git log` コマンドで全変更ファイルを対象に実行（ファイル内容を読まないため影響なし）
- 他の全メタ分析は現行通り実行（入力が大きくなるだけ）

## 3. SCALE_CONTEXT ブロック形式

Phase 0 の出力に以下を追記する:

```markdown
**スケール情報**:
- スコープファイル数: 45
- チャンク計画: 3チャンク（チャンク1: 20件, チャンク2: 20件, チャンク3: 5件）
- モノレポ検出: あり（packages: api, web, shared-types）
- アクティブフィルタ: --path src/auth
- 推定所要時間: 65-90分
```

## 4. 失敗回復プロトコル

### F-5: チャンク処理中にエージェントがタイムアウト

**症状**: チャンク N のエージェントが空の出力を返す、または応答が極端に遅い。
**回復手順**:
1. そのチャンク（20件）をさらに10件×2チャンクに分割する
2. 分割したチャンクを再度同じエージェント指示で処理する
3. 成功した場合はfindings を前チャンクの蓄積に追加して継続
4. 再度失敗した場合: そのファイルを「手動レビュー推奨」としてレポートに記録し、残りのチャンクを続ける

### F-6: モノレポのパッケージ境界が不明確

**症状**: ファイルがどのパッケージに属するか判定できない（ルートファイル等）。
**回復手順**:
1. ルートファイル（`package.json`, `turbo.json`, `Cargo.toml` 等）は「インフラファイル」として分類
2. パス不明のファイルは通常のディレクトリ順グループ化にフォールバック
3. Phase 0 出力に `[MONOREPO:BOUNDARY_AMBIGUOUS]` を記録

### F-7: --path / --package フィルタが 0 件にマッチ

**症状**: 指定したパスに変更ファイルが存在しない。
**回復手順**:
1. エラーメッセージを表示: `指定パス "${PATH}" に変更ファイルが見つかりません`
2. 候補を提示:
   ```bash
   git diff --name-only HEAD~1 HEAD | head -20
   ```
3. フィルタなしで続行するかユーザーに確認する

### F-8: --package 名がモノレポレジストリに未存在

**症状**: 指定したパッケージ名がどのパッケージにもマッチしない。
**回復手順**:
1. エラーメッセージを表示: `パッケージ "${PACKAGE}" が見つかりません`
2. 検出済みパッケージ一覧を表示
3. フィルタなしで全変更ファイルを対象に続行する
